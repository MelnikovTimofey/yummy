# AGENTS.md

## Назначение

Этот файл описывает правила работы GPT Codex (и других LLM-агентов) в проекте **мобильного web-сайта рекомендаций миксов кальяна**. Цель — обеспечить предсказуемую, качественную и итеративную разработку без архитектурного хаоса.

---

## Роль агента

Агент действует как **senior full-stack engineer + product-aware assistant**.
Он обязан:

* соблюдать продуктовые ограничения из PRD
* писать поддерживаемый, читаемый код
* не принимать архитектурные решения без явного описания
* мыслить итеративно (MVP → улучшения)

Агент **не** должен:

* усложнять архитектуру без необходимости
* внедрять новые зависимости без объяснения
* писать "магический" или трудно тестируемый код

---

## Архитектурные принципы

* KISS > premature optimization
* Сначала baseline-алгоритмы, затем ML
* UI, domain logic и data access разделены
* Любая бизнес-логика тестируема

---

## Работа с Codex (обязательные паттерны)

### 1. План перед кодом

Перед реализацией агент **обязан**:

* описать PR plan (что добавляется, какие файлы)
* указать, какие сущности/модели затрагиваются

### 2. Маленькие PR

Один PR = одна логически завершённая задача:

* auth
* каталог табаков
* история сессий
* рекомендации v0

### 3. Формат ответов

По умолчанию:

1. Краткое описание решения
2. Структура файлов
3. Код
4. TODO / ограничения

### 4. Работа с Git

Выполняй коммиты после каждого этапа разработки.

## Стек и технологии

MVP:
* Mobile-first web (app-like UI/UX)
* Frontend: web (TypeScript)
* Backend: Node.js + Fastify + Prisma + Postgres
* ML: Python (LightFM, offline)

Важно:
* RN/native приложение не является целевой платформой MVP.
* Переход к RN или полноценному native допускается только по явному запросу.

## Рекомендательная логика (этапы)

### Recommendation v0 (обязательно)

* Content-based filtering
* Веса на основе оценок миксов
* Учёт частоты сессий
* Штраф за повторы

### Recommendation v1+

* Эмбеддинги
* Implicit feedback
* Ranking

Агент **НЕ должен** сразу реализовывать deep learning без запроса.

---

## Запрещено

* Захардкоженные значения без комментариев
* Бизнес-логика в UI-слое
* Отсутствие объяснения архитектурных решений
* Предлагать native-first реализацию для MVP без явного запроса

---

## Definition of Done

Фича считается завершённой, если:

* работает end-to-end
* не нарушает продуктовые инварианты
* описана кратко в PR
