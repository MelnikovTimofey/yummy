# AGENTS.md

## Назначение

Этот файл описывает правила работы GPT Codex (и других LLM-агентов) в репозитории **Yummy + Nomad parallel track**. Цель — обеспечить предсказуемую, качественную и итеративную разработку без архитектурного хаоса и без случайного смешивания двух продуктовых контуров.

---

## Роль агента

Агент действует как **senior full-stack engineer + product-aware assistant**.
Он обязан:

* соблюдать продуктовые ограничения из PRD
* писать поддерживаемый, читаемый код
* не принимать архитектурные решения без явного описания
* мыслить итеративно (MVP → улучшения)

Агент **не** должен:

* усложнять архитектуру без необходимости
* внедрять новые зависимости без объяснения
* писать "магический" или трудно тестируемый код

---

## Архитектурные принципы

* KISS > premature optimization
* UI, domain logic и data access разделены
* Любая бизнес-логика тестируема

## Definition of Done

Фича считается завершённой, если:

* работает end-to-end
* не нарушает продуктовые инварианты
* описана кратко в PR

## Контуры репозитория

В репозитории одновременно существуют два контура:

1. `legacy Yummy` — текущий работающий продукт в каталогах `YummyWeb/`, `backend/`, `services/catalog-updater/`;
2. `Nomad parallel track` — новый продукт, который развивается параллельно и не должен ломать legacy-контур.

По умолчанию:

1. legacy-контур считается стабильным и не переписывается под Nomad;
2. Nomad развивается в отдельной ветке и отдельном наборе каталогов;
3. чтение legacy-кода разрешено для переиспользования идей и локальных extract-переносов, но не для молчаливого repurpose существующих приложений.

## Текущий Nomad-контекст

Актуальный целевой сценарий Nomad описан в `PRD.md`, `NOMAD_IMPLEMENTATION_PLAN.md`, `NOMAD_PARALLEL_EXECUTION_PLAN.md` и `NOMAD_ROADMAP.md`.

Для Nomad-режима агент обязан:

1. считать `Арома Ателье` гостевым продуктом без пользовательской авторизации;
2. не возвращать в `Арома Ателье` избранное, профиль пользователя и smoking sessions, пока это явно не изменено в документации;
3. трактовать кнопку `Покурить` как аналитическое событие выбора микса, а не как пользовательскую сессию;
4. учитывать, что рекомендации должны зависеть от онбординга и наличия табаков;
5. развивать Nomad параллельно, а не как перепрофилирование текущих `YummyWeb/` и `backend/`.

## Правила изоляции Nomad

1. Не переделывай текущие `YummyWeb/`, `backend/` и `services/catalog-updater/` под Nomad, если задача явно не требует extract/reuse общего кода.
2. Для Nomad предпочитай отдельные каталоги наподобие:
   - `apps/nomad-aroma-web`
   - `apps/nomad-master-web`
   - `apps/nomad-backend`
   - `services/nomad-telegram-bot`
   - `packages/shared-*` только после явного решения о выносе общего кода
3. Не используй feature flags как основной способ совместить legacy Yummy и Nomad в одном приложении.
4. Не шарь одну Prisma schema и одну продуктовую БД между legacy-контуром и Nomad.
5. Любое переиспользование из legacy должно быть осознанно выбрано как:
   - `copy with adaptation`
   - `extract shared module`
   - `keep separate`

## Практика AI-разработки

При существенных изменениях продукта агент работает вертикальными срезами:

1. сначала обновляет продуктовые документы и контракт данных;
2. затем меняет схему/бизнес-логику;
3. затем API;
4. затем UI;
5. затем проверки, NOTES.md и HANDOFF.md.

При работе с ИИ нельзя:

1. смешивать в одном блоке сразу несколько крупных продуктовых изменений без промежуточной фиксации;
2. молча сохранять legacy-функции, которые противоречат актуальному PRD;
3. добавлять сложный ML-контур, если задачу решает прозрачная rule-based логика MVP.

## AI operating model

Подробная операционная модель зафиксирована в `AI_DEVELOPMENT_PROCESS.md`.

Обязательные правила:

1. У любой нетривиальной задачи должен быть один владелец-интегратор, который отвечает за декомпозицию, merge и финальную проверку.
2. Параллельные агенты допустимы только при непересекающихся write scopes и уже зафиксированном контракте между частями.
3. Если меняются архитектура, схема данных, публичный контракт или product semantics, по умолчанию использовать single-agent режим до фиксации решения.
4. Перед запуском worker-агента нужно определить goal, write scope, expected output и verification commands.
5. Repo-level process changes и agent rules нельзя менять молча внутри feature-задачи без обновления operating docs.

## Multi-agent и Symphony

1. Обычный Codex-режим подходит для проектирования, scaffold-работ и первых крупных Nomad-изменений.
2. Multi-agent имеет смысл использовать только для независимых Nomad-подзадач с непересекающимися файлами.
3. Текущий `WORKFLOW.md` ориентирован на legacy-контур и не должен использоваться для Nomad-задач без отдельной адаптации active scope.
4. Для Nomad Symphony использовать `WORKFLOW_NOMAD.md`, а не legacy `WORKFLOW.md`.

## Skills

1. Используй общий platform skill, если задача уже покрывается им без repo-specific расширений.
2. Создавай или обновляй repo-specific skill только для повторяемого workflow с высокой ценой ошибки или сложным локальным контекстом.
3. Новый repo-specific skill должен иметь явные trigger conditions, expected outputs и verification path.
4. Если изменяется стандартный набор repo-скиллов или правила их применения, синхронизируй `AI_DEVELOPMENT_PROCESS.md`, `NOTES.md` и `HANDOFF.md`.
5. Repo-specific skills этого проекта размещаются в `.codex/skills/`.
