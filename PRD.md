# PRD — Мобильный web-сайт рекомендаций миксов кальяна

## 1. Обзор продукта

Мобильный web-сайт помогает пользователю подбирать **персонализированные миксы кальяна** на основе:

* вкусовых предпочтений
* истории курения
* оценок миксов и сессий курения

Фокус продукта — **персональные рекомендации**, а не социальная сеть.

MVP-платформа: mobile web (в браузере), с интерфейсом и навигацией в стиле мобильного приложения.

---

## 2. Целевая аудитория

* Пользователи, регулярно курящие кальян
* Любители экспериментов со вкусами
* Владельцы домашних кальянов и гости кальянных

---

## 3. Цели продукта

### Бизнес / продукт

* Увеличить частоту экспериментов с новыми 
* Повысить удовлетворённость рекомендациями

### Пользовательские

* Быстро понять, "что сегодня покурить"
* Не повторять неудачные миксы
* Запоминать удачные сочетания

---

## 4. Основные фичи (MVP)

### 4.1 Онбординг (cold start)

* Выбор вкусовых профилей:
  * сладкий
  * кислый
  * пряный
  * свежий
  * десертный
  * табачный
* Выбор любимых брендов
* Выбор нелюбимых профилей (опционально)

---

### 4.2 Аутентификация

* Через почту (passwordless magic link)
* Анонимный режим **не поддерживается**

---

### 4.3 Каталог табаков

Каждый табак содержит:

* название
* производитель
* вкусовые профили
* крепость / линейка (в диапазоне от нуля до 10)
* цвет (генерируется по вкусовому профилю, без изображений)

Функции:

* просмотр
* фильтрация
* поиск

---

### 4.4 Миксы

#### Сущность Mix

* название
* список компонентов (табак + пропорция)
* автор (user)
* дата создания

#### Правила

* Микс переиспользуем
* Один микс может иметь много сессий

---

### 4.5 История курения (Smoking Sessions)

Сессия содержит:

* дату
* микс
* локация - дом, название кальянной

---

### 4.6 Оценки

1. Сессия курения оценивается пользователем
2. Микс оценивается пользователем. Если пользователь не оценивал микс - оценка формируется как среднее от всех сессий курения

---

### 4.7 Рекомендации

Кнопка "Рекомендовать" формирует список миксов:

* на основе PreferenceProfile
* с учётом истории и оценок

Логика выдачи на экране рекомендаций (приоритеты):

1. Если есть персональная модель пользователя (индивидуальный скоринг и актуальный предрасчёт), экран показывает рекомендации модели.
2. Если персональная модель недоступна или не вернула достаточное число элементов, используется fallback-ранжирование топов, где одновременно учитываются:
   * средняя оценка микса;
   * количество сессий курения с этим миксом.
   Оба параметра влияют на итоговый порядок выдачи (композитный score).
3. Если данных по оценкам недостаточно (cold start), используется fallback:
   * популярные миксы по количеству сессий;
   * мягкая персонализация по онбордингу (любимые профили/бренды), если она доступна;
   * диверсификация (не показывать подряд однотипные миксы).

Механика экрана рекомендаций наследуется от приложений для знакомств, в которых пользователь свайпает в право и влево предложенные варианты. Влево - пользователь желает посмотреть рекомендаци далее (также), вправо - появляется меню с действиями "Добавить в сессию курения", "Оценить микс"

---

### Меню

Разделы меню "внизу" должны быть в следующем порядке:
1. Миксы
2. Сессии курения
3. Каталог
4. Рекомендации
5. Профиль

Порядок отражает пользовательскую цель: быстро выбрать микс и добавить его в сессию.

---

### 4.8 UI/UX-инварианты v0.1 (фиксируем комментарии из разработки)

1. Интерфейс полностью на русском языке.
2. Mobile web должен учитывать safe area (верх/низ) на iPhone через CSS env(safe-area-inset-*); контент не должен залезать под Dynamic Island/камеру и системные элементы.
3. Все экраны должны занимать всю высоту viewport мобильного браузера; нижний таб-бар фиксируется снизу и не перекрывает контент списков.
4. На полноэкранных экранах создания/выбора обязательно наличие явного действия "Назад" (или "Закрыть").
5. Экран "Создать микс" является отдельным экраном; в списке миксов отображается только кнопка "Создать микс".
6. Экран создания сессии не должен содержать длинный встроенный список миксов; выбор микса выполняется на отдельном экране выбора.
7. В карточках миксов и сессий должны отображаться две оценки: пользовательская и средняя.
8. Нижние табы не должны иметь горизонтального обрезания подписей.
9. Интерфейс ориентирован на touch-взаимодействие (кнопки/контролы удобны для пальца).
10. MVP должен работать без установки приложения: открытие по URL в мобильном браузере.

### 4.9 База данных табаков и миксов

1. Должа быть сформирована база данных табаков популярных производителей (deus, darkside, must have)
2. Должна быть сформирована база данных миксов (100 - 200)
3. Источники данных — только официальные сайты брендов


## 5. Алгоритм рекомендаций

### v0.1 (MVP)

* ML‑модель LightFM (hybrid)
* Обучение на взаимодействиях (сессии + оценки)
* Оффлайн предрасчёт рекомендаций и хранение в БД
* Приоритет выдачи:
  1. персональная модель;
  2. fallback по композитному скору (оценка + число сессий);
  3. cold-start fallback (популярность + онбординг + диверсификация)
* Для fallback-ранжирования без персональной модели используется формула, где и средняя оценка, и частота сессий являются обязательными факторами сортировки

### v1+

* Расширение модели (доп. фичи, текстовые теги)
* Улучшение pipeline/персонализации

---

## 6. Бэкенд и инфраструктура

* Open‑source стек
* Развёртывание локально и на сервере
* Компоненты:
  * Frontend: mobile web клиент (app-like UI)
  * Postgres
  * Node.js + Fastify + TypeScript (API + бизнес‑логика)
  * Python‑сервис LightFM (оффлайн обучение)
  * Mailpit для локальной почты

---

## 7. Нефункциональные требования

* Быстрый отклик рекомендаций (<300 мс)
* Оффлайн‑предрасчёт для рекомендаций
* Масштабируемость данных

---

## 8. Ограничения

* Контент 18+
* Без социальной ленты
* Без рекомендаций по никотину / здоровью

---

## 9. Риски

* Малое количество данных на старте
* Субъективность оценок

Митигируется:

* cold start через популярные миксы + диверсификацию
* постепенное накопление данных

---
