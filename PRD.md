# PRD — Nomad Aroma Atelier + Master

## 1. Обзор продукта

Продукт состоит из трёх связанных контуров для лаундж-бара Nomad:

1. `Арома Ателье` — гостевой mobile web без авторизации.
2. `Мастер` — staff web для кальянных мастеров и администратора.
3. `Telegram-бот Nomad` — канал распространения ежедневного кода доступа для персонала.

Цель MVP: быстро доводить гостя от QR-кода до понятной карточки микса, которую он показывает кальянному мастеру, а персоналу дать простой и поддерживаемый контур управления наличием, миксами, рейлами и статистикой.

## 2. Цели продукта

### 2.1 Бизнес

* повысить конверсию из сканирования QR в нажатие `Покурить`;
* сократить время подбора микса гостю;
* держать рекомендации в актуальном состоянии за счёт учёта реального наличия табаков;
* дать Nomad инструмент ежедневной операционной работы без ручного хаоса.

### 2.2 Пользовательские

#### Гость

* быстро понять, как работает сервис;
* выбрать, что покурить, без регистрации;
* получить готовую карточку микса, которую можно показать мастеру;
* при желании оценить микс.

#### Кальянный мастер

* быстро отметить, что есть в наличии;
* добавлять новые табаки и миксы;
* собирать и редактировать рейлы;
* видеть, что чаще выбирают гости.

#### Администратор

* управлять всем каталогом и staff-доступом;
* контролировать ежедневный код и статистику;
* иметь единый источник правды по каталогу и аналитике.

## 3. Роли и доступ

### 3.1 Aroma Atelier

Доступ без пользовательской авторизации.

Гость проходит:

1. QR-старт;
2. age gate `18+`;
3. ввод ежедневного кода, который ему сообщает сотрудник Nomad;
4. знакомство;
5. онбординг по вкусовым предпочтениям;
6. рекомендации / главная / каталог / карточка микса;
7. действие `Покурить`;
8. оценка микса.

### 3.2 Master

Доступ только по логину и паролю.

Поддерживаются две учётные записи:

1. `admin` — полный доступ владельца системы;
2. `nomad` — staff-доступ для кальянных мастеров.

На MVP права можно реализовать двумя ролями:

1. `admin` — полный CRUD + управление staff + служебные настройки;
2. `staff` — работа с наличием, миксами, рейлами и просмотром дашбордов.

### 3.3 Telegram-бот

Бот доступен только сотрудникам Nomad и рассылает ежедневный код. Бот не является guest-интерфейсом.

## 4. Ограничения и инварианты

1. Интерфейс полностью на русском языке.
2. В `Арома Ателье` нет пользовательской авторизации.
3. В `Арома Ателье` нет избранного, профиля пользователя и сессий курения.
4. История персонального пользователя не хранится; персонализация только в рамках текущего гостевого прохождения.
5. Кнопка `Покурить` не создаёт пользовательскую smoking-session, а пишет аналитическое событие выбора микса.
6. Механика оценок сохраняется.
7. Перед доступом к основному функционалу обязателен экран подтверждения `18+`.
8. В рекомендациях и каталоге должны учитываться только табаки, доступные по текущему наличию, если сценарий явно не допускает показ недоступных позиций.
9. Для каталога табаков использовать разделение: `flavorProfiles` — категории, `flavors` — вкусы, `flavorTags` — только мета-теги.
10. Визуальная база продукта использует бренд-направление Nomad. Базовый primary color: `#551817`.

## 6. Рекомендательная логика

### 6.1 Основа MVP

Персональная рекомендация формируется на основе:

1. ответов текущего гостя в онбординге;
2. состава микса;
3. актуального наличия табаков на кухне.

### 6.2 Правила MVP

1. Если хотя бы один компонент микса отсутствует в наличии, такой микс не должен попадать в основную гостевую рекомендацию.
2. Онбординг должен быть коротким и приводить к полезной первой выдаче за 1-2 минуты.
3. Рекомендация должна быть объяснимой на уровне продукта: вкусовые профили, вкусы, крепость, настроение.
4. При отсутствии сильного совпадения пользователь должен быстро перейти к рейлам или каталогу.

### 6.3 Без ML на старте

На MVP не требуется ML-персонализация. Достаточно rule-based ранжирования, которое использует:

* совпадение по `flavorProfiles`;
* совпадение по `flavors`;
* штраф за конфликтующие ответы;
* бонус за статистическую популярность;
* бонус/сортировку по оценкам.

ML допустим только как следующий этап после накопления стабильных событий.

## 7. Рейлы

Рейл обязан иметь `type`.

Типы рейлов:

1. `statistical` — статистические рейлы.
2. `prepared` — предзаготовленные рейлы.
3. `curated` — рейлы, сформированные кальянными мастерами.

### 7.1 Обязательные statistical-рейлы MVP

1. `Топ по кнопке "Покурить"`
2. `Топ по оценкам`

### 7.2 Правила показа

1. На `Главной` Aroma Atelier пользователь видит рейлы, если рекомендации не подошли или он хочет продолжить выбор.
2. Предзаготовленные и curated-рейлы управляются через `Мастер`.
3. Statistical-рейлы пересчитываются автоматически по аналитическим событиям.

## 8. Экранная модель Aroma Atelier

### 8.1 `18+`

Экран подтверждения возраста перед любым дальнейшим действием.

### 8.2 Ввод кода

Пользователь вводит ежедневный код доступа, полученный от сотрудника Nomad.

### 8.3 Знакомство

Набор карточек, которые объясняют сценарий использования сервиса.

### 8.4 Онбординг

Короткий сценарий выбора предпочтений:

* профили вкуса;
* конкретные вкусы;
* при необходимости дополнительные фильтры, если они помогают быстрее получить релевантную выдачу.

Результат онбординга: список карточек рекомендованных миксов.

### 8.5 Главная

Список рейлов, доступных гостю после рекомендаций.

### 8.6 Каталог

Самостоятельный выбор микса с поиском и фильтрами.

### 8.7 Карточка микса

Карточка обязана содержать:

1. название микса;
2. состав и пропорции;
3. вкусовые профили;
4. описание;
5. CTA `Покурить`;
6. возможность поставить оценку.

После нажатия `Покурить` открывается чистая карточка-подтверждение, удобная для показа кальянному мастеру.

## 9. Экранная модель Master

### 9.1 Авторизация

Логин/пароль для `admin` и `nomad`.

### 9.2 Дашборд по сессиям курения

На MVP трактуется как дашборд по аналитическим событиям выбора миксов, а не по реальным пользовательским аккаунтам.

Базовые метрики:

* нажатия `Покурить` по дням;
* топ миксов;
* top / anti-top профили вкусов;
* конверсия из онбординга в `Покурить`, если событие онбординга сохраняется.

### 9.3 Инвентаризация

1. отметка `в наличии / не в наличии`;
2. добавление новых табаков;
3. быстрый поиск по существующему каталогу.

### 9.4 Дашборд по инвентаризации

Разрезы:

* всего табаков;
* в наличии / не в наличии;
* по производителям;
* по вкусовым профилям;
* по крепости.

### 9.5 Менеджер миксов

CRUD для миксов и их компонентов.

### 9.6 Менеджер рейлов

CRUD для рейлов всех типов и управление наполнением.

## 10. Telegram-бот

MVP-функции:

1. ежедневная генерация или публикация кода;
2. отправка кода в staff-чат или ограниченному списку пользователей;
3. ручной запрос текущего кода командой;
4. журнал последнего активного кода.

## 11. События и аналитика

Минимальный набор событий MVP:

1. `qr_opened`
2. `age_confirmed`
3. `daily_code_submitted`
4. `daily_code_verified`
5. `intro_completed`
6. `onboarding_completed`
7. `recommendation_opened`
8. `mix_card_opened`
9. `smoke_cta_clicked`
10. `mix_rated`

`smoke_cta_clicked` — основной продуктовый сигнал для статистических рейлов и дашбордов.

## 12. UI/UX-инварианты

1. Mobile-first для `Арома Ателье`.
2. Быстрый сценарий выбора без длинных обязательных форм.
3. Явные CTA и минимальное число шагов до первой рекомендации.
4. Визуальное направление должно быть связано с Nomad: тёплые тёмные акценты, премиальный lounge-character.
5. Интерфейс мастера допускает более утилитарный стиль, но должен оставаться в общей палитре.

## 13. Нефункциональные требования

1. Все ключевые бизнес-правила тестируемы на уровне backend/domain.
2. Гостевой код должен иметь срок действия и возможность ежедневной ротации.
3. Любое изменение наличия должно отражаться в рекомендациях без ручной синхронизации данных между сервисами.
4. Система должна корректно работать без ML и без пользовательских аккаунтов гостей.
