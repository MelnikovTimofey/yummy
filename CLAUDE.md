# CLAUDE.md

Инструкция для Claude Code и субагентов в репозитории **Nomad**.

## 1. Проект

Репозиторий содержит контур Nomad. Каталоги: `apps/nomad-aroma-web`,
`apps/nomad-master-web`, `apps/nomad-backend`, `services/nomad-telegram-bot`,
smoke-тесты в `tests/nomad-smoke`.

Исходный продукт **legacy Yummy** вынесен в архивный репозиторий
[MelnikovTimofey/yummy](https://github.com/MelnikovTimofey/yummy) и здесь больше
не присутствует. Состояние монорепо до split — tag `pre-legacy-split`.

Production-ветка репозитория — `main`. Соглашение по именам рабочих веток см. §5.

Продукт Nomad (детали — в `PRD.md`, `docs/nomad/`):

- `Арома Ателье` (`nomad-aroma-web`) — гостевой продукт **без авторизации**: онбординг
  → рекомендации миксов → кнопка `Покурить`.
- `Мастер` (`nomad-master-web`) — backoffice для роли `nomad`: инвентарь, миксы, rails.
- `nomad-backend` — Node/TypeScript + Prisma + Postgres, интеграция с `htreviews.org`.

## 2. Принципы (KISS)

- Простое решение вперёд преждевременных абстракций. Три похожие строки лучше ранней
  абстракции. Не проектировать под гипотетическое будущее.
- Не добавлять обработку ошибок и валидацию для невозможных сценариев; валидировать
  только границы системы (пользовательский ввод, внешние API).
- Без backwards-compat хаков и feature-флагов, если можно просто изменить код.
- Работать вертикальными срезами: контракт данных → схема/логика → API → UI → проверки.
- Не смешивать в одном изменении несколько крупных продуктовых решений.
- Интерфейс и файлы `README.md` — на русском языке.
- Комментарии — только когда «почему» неочевидно; не описывать «что» делает код.

## 3. Разработка через тесты (TDD)

- Для нетривиальной логики: сначала падающий тест, затем код, затем рефакторинг.
- Backend-тесты: `cd apps/nomad-backend && npm test` (нужен Postgres —
  `npm run db:start`). Любая бизнес-логика должна быть тестируемой.
- Перед PR обязателен зелёный gate по затронутым поверхностям:
  - `apps/nomad-backend` → `npm test && npm run build`;
  - `apps/nomad-master-web` → `npm test && npm run build`;
  - `apps/nomad-aroma-web` → `npm run build`;
  - `services/nomad-telegram-bot` → `npm test && npm run build`;
  - UI/интеграция → `cd tests/nomad-smoke && npm run smoke`.
- Если проверку не удалось запустить (например, нет окружения) — явно сказать об этом,
  а не выдавать за успех.

> 🚨 **ЖЁСТКОЕ ПРАВИЛО — smoke после правок фронта:** любое изменение в
> `apps/nomad-master-web/src/**` или `apps/nomad-aroma-web/src/**`, которое
> затрагивает разметку, текст, ARIA-ролы, h1/h2/heading'и, label'ы, табы или
> навигацию — **обязано** в том же PR пройти через `cd tests/nomad-smoke &&
> npm run smoke` (а при невозможности запустить — отдельно перечислить, какие
> assertion'ы в `tests/nomad-smoke/tests/*.spec.ts` могут поплыть, и
> обновить их). Триггер этой записи: в апреле–мае 2026 master-web прошёл
> два UX-рефактора (PR #12, #14) с переименованием h1, smoke на `main`
> оставался красным на каждом последующем PR, что блокировало merge. Не
> мерджить «оранжевый» smoke со списком «починим потом» — drift лечится в
> том же PR, что породил его. Если smoke упал на CI после PR-mainline и
> причина — UI drift, прежний фронт-PR считается недоставленным.

## 4. Агенты и роли

Кастомные субагенты — в `.claude/agents/`:

- **planner** — анализ задачи, декомпозиция, формирование GitHub issue, выбор режима.
- **backend-dev** — `apps/nomad-backend` (+ Prisma, bot), TDD.
- **frontend-dev** — `apps/nomad-aroma-web` и `apps/nomad-master-web`, TDD.
- **qa-tdd** — тесты-первыми, smoke, верификация перед PR.

Плюс встроенные `Explore` (поиск по коду) и `Plan` (проектирование).

Правила мультиагентной работы:

- Параллельные агенты — только при **непересекающихся write-scope** и уже
  зафиксированном контракте между частями.
- Если меняются архитектура, схема данных, публичный контракт или продуктовая
  семантика — работать одним агентом до фиксации решения.
- У нетривиальной задачи один владелец-интегратор: декомпозиция, сборка, проверка.
- Перед запуском субагента задать ему цель, write-scope, ожидаемый результат и
  команды верификации.

Углублённые чек-листы по поверхностям — в `.claude/skills/`.

## 5. Процесс: issue → ветка → PR

> 🚨 **ЖЁСТКОЕ ПРАВИЛО — НЕ ИГНОРИРОВАТЬ:** любые изменения файлов делать
> ТОЛЬКО в отдельной ветке `feature/<slug>`, `bug/<slug>` или
> `chore/<slug>` от **свежего** `main`. Прямые правки в `main`
> (Edit/Write/коммит) запрещены — даже для «мелочей», Dockerfile, конфигов,
> документации. Перед первым `Edit`/`Write` в сессии:
> 1. `git checkout main && git pull origin main` — обновить локальный `main`
>    до origin (иначе ветка отпочкуется от устаревшего состояния и при PR
>    будут конфликты с уже смерженными работами других).
> 2. `git checkout -b feature/<slug>` (или `bug/<slug>` / `chore/<slug>`) —
>    отпочковать новую ветку. Выбор префикса — см. §5.2.
> 3. `git rev-parse --abbrev-ref HEAD` — убедиться, что не на `main`.
>
> Никогда не продолжать работу на старой `feature/*` / `bug/*` / `chore/*`
> ветке, если её PR уже смерджен: для новой задачи — новая ветка от свежего
> `main`. Если сомневаешься в названии слага — спросить пользователя, а не
> уходить в `main`.

> 🚨 **ЖЁСТКОЕ ПРАВИЛО — параллельные ветки не мешают друг другу:** когда
> одновременно идёт больше одной задачи, каждая живёт в **отдельном
> git worktree**, а не в переключении веток внутри одной рабочей копии.
> Контракт «одна сессия — одна ветка — одна рабочая копия»:
>
> 1. **Один срез = один worktree.** Параллельная задача создаётся через
>    `git worktree add ../nomad-<slug> -b feature/<slug>` (или `bug/` /
>    `chore/`) от свежего `main`. Не переключать ветки в основной копии,
>    пока в ней есть незакоммиченные правки от другой задачи — переезд
>    утянет их в новую ветку и смешает scope.
> 2. **Anchor-чек на старте каждой сессии.** Перед первым `Edit`/`Write`:
>    `pwd && git rev-parse --abbrev-ref HEAD && git status --short`. Если
>    ветка не та, что ожидалась, или в `git status` есть «чужие»
>    untracked/modified файлы из соседней задачи — стоп, разобраться, не
>    писать.
> 3. **Не переключать ветку внутри сессии.** Если задача меняется в ходе
>    разговора — закрыть текущую (commit или явный stash с пометкой) и
>    начать новую сессию от свежего `main`. `git checkout` другой ветки
>    посреди сессии — запрещён.
> 4. **Edit/Write — только относительно cwd сессии.** В worktree-сессии
>    абсолютный путь в основную копию (`/Users/.../yummy/...` вне своего
>    worktree) запишет в `main`-копию, а не в worktree, и правка
>    «потеряется» относительно ветки. Никаких абсолютных путей за
>    пределы своего worktree; внутри worktree — путь относительно его
>    корня.
> 5. **Закрывать worktree после merge.** `git worktree remove
>    ../nomad-<slug>` сразу после merge PR — иначе worktrees копятся и
>    теряется картина «кто чем занят». `git worktree list` — single
>    source of truth о параллельных сессиях; перед стартом новой задачи
>    свериться с ним и убедиться, что слот свободен.

1. Нетривиальная задача начинается с GitHub issue
   (`.github/ISSUE_TEMPLATE/nomad-feature.yml` / `nomad-bug.yml` / `nomad-ops.yml`).
   Issue фиксирует bounded context, scope, критерии и проверки.
2. Ветка от **свежего** `main` (после `git pull origin main`) по соглашению:
   - `feature/<slug>` — новый функционал, рефакторинг с продуктовым
     эффектом, обновление процесса/документации, новая поверхность;
   - `bug/<slug>` — исправление бага (видимого пользователю или ломающего
     инвариант);
   - `chore/<slug>` — техническая уборка без продуктового эффекта:
     удаление dead-code, cleanup осиротевших ссылок, апдейт зависимостей,
     CI/lint/format-tweaks, переименование внутренних идентификаторов.
     Не подходит для chore: схема данных, публичный контракт, auth —
     эти изменения идут через `feature` и попадают под human review
     (см. `.github/NOMAD_REVIEW_POLICY.md`).

   Если сомневаешься между `feature` и `chore` — спрашивай. Правило
   большого пальца: если PR-описание начинается со слов «удалить /
   убрать / починить опечатку / обновить версию» и не меняет
   пользовательского поведения — это `chore`. Если меняет поведение
   или добавляет/улучшает функциональность — `feature`.

   `<slug>` в kebab-case, при необходимости с issue-номером
   (`feature/123-master-shell`). Один вертикальный срез на ветку.
3. TDD-цикл и зелёный verification gate (раздел 3).
4. PR в `main` по `.github/pull_request_template.md`:
   **1 PR = 1 bounded context**, без смешения схемы/auth/UI/инфры.
5. Self-review до merge. Изменения схемы/auth/env/контракта/процесса требуют
   человеческого ревью — см. `.github/NOMAD_REVIEW_POLICY.md`.
6. Коммит после каждого логического блока.

## 6. Отношения с legacy Yummy

- Legacy-код живёт отдельно: [MelnikovTimofey/yummy](https://github.com/MelnikovTimofey/yummy).
  В этом репозитории его нет и сюда он не возвращается.
- Не копировать legacy-каталоги (`Yummy/`, `YummyWeb/`, `backend/`, `ml/`,
  `services/catalog-updater/`) обратно в `apps/`/`services/`.
- Переиспользование идей из legacy — осознанный выбор: чтение по ссылке,
  `copy with adaptation` или `extract shared module`. Не зеркалить файлы.

## 7. Продуктовые инварианты Nomad

1. `Арома Ателье` — гостевой контур без пользовательской авторизации; не возвращать
   избранное, профиль и smoking sessions, пока это явно не изменено в `PRD.md`.
2. `Покурить` — аналитическое событие выбора микса, а не пользовательская сессия.
3. Рекомендации зависят от онбординга и наличия табаков в инвентаре.
4. Таксономия каталога табака: `flavorProfiles` (категории), `flavors` (вкусы),
   `flavorTags` (только мета-теги) — три разных поля, не смешивать.

## 8. Документация

- `HANDOFF.md` — rolling-журнал: обновлять при завершении логического блока, при
  изменении правил/процесса/UX, чтобы решение оставалось discoverable.
- При изменении правил агента — обновлять этот `CLAUDE.md`.
- Не плодить новые md-артефакты под каждое микро-изменение.

## Главное правило

**Если есть сомнение — сначала задать вопрос или предложить вариант, а не молча
принимать решение.**

**И никогда не писать в `main` напрямую — всегда `feature/<slug>`,
`bug/<slug>` или `chore/<slug>` (см. §5).**
